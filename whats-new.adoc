=== 數位顧問



==== 健康檢查 GraphQL 遷移和資料準確性改進

健康檢查模組已增強，資料準確性更高，版本建議也與現行標準保持一致。作為此版本更新的一部分，該模組已從公共 API 遷移至 GraphQL，從而提升了效能，提供更靈活的查詢方式及長期支援。



===== 推薦軟體

健康檢查模組中推薦的軟體增強功能現在可以提供更準確、更一致的資料。

* 機櫃和磁碟機數量在韌體版本顯示下會顯示每個系統的準確值，不再包含 HA 配對數量。
* 現在，最低版本和最新版本建議是根據叢集的有效版本在叢集層級計算的，符合現行標準——以前，這些建議是在節點層級計算的。




===== AutoSupport 資料依賴性

由於後端改進，AutoSupport 採用率和建議軟體視圖均依賴最新的 AutoSupport 資料。這可能會以下列方式影響您目前的視圖：

[cols="30,70"]
|===
| 情境 | 受影響的系統 


| 未顯示資料 | 自 2025 年 10 月以來未傳送 AutoSupport 的系統 


| 過時的值 | 過去 30 天內未傳送 AutoSupport 的系統 
|===
在上述受影響的系統上重新觸發 AutoSupport，以還原健康檢查視圖中的最新資料。

=== 備份和復原



==== Kubernetes 工作負載增強功能

在此版本中、Kubernetes 工作負載引入了以下增強功能：

* *Trident Protect 26.06*：此版本包含 Trident Protect 26.06。
* *恢復期間跳過應用程式建立*：現在您可以在復原作業期間跳過自動建立應用程式自訂資源物件。這樣可以減少應用程式清單中不必要的條目。
* *還原 VM 持久性磁碟區請求*：還原 VM 型應用程式時，現在可以還原個別持久性磁碟區請求，而無需還原整個虛擬機器，當特定磁碟或磁碟區受到損毀或資料遺失影響時，可加快復原速度。
* *作業監控改進*：作業監控功能現已增強，可更清楚顯示 Kubernetes 保護作業的狀態。您可以查看作業和子作業的詳細狀態資訊，以及每個作業使用的目標儲存。


有關保護 Kubernetes 工作負載的詳細資訊，請參閱 https://docs.netapp.com/us-en/data-services-backup-recovery/br-use-kubernetes-protect-overview.html["保護 Kubernetes 工作負載概述"]。

=== 災難復原



==== Kubernetes 更新

災難恢復針對基於 Kubernetes 的工作負載引入了以下變更：



===== 測試 Kubernetes 工作負載的容錯移轉支援

災難復原功能現已支援對 Kubernetes 工作負載的故障轉移流程進行測試。透過測試故障轉移，您可以驗證災難復原計劃。測試完成後，您可以清理測試故障轉移，系統會自動清理測試資源，從而降低系統開銷並確保環境的一致性。

如需詳細資訊、請參閱 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/failover.html#test-the-failover-process["測試容錯移轉程序"^]。



===== 排程測試容錯移轉

現在，您可以透過排程測試容錯移轉來自動化災難恢復測試。您可以設定定期測試容錯移轉的排程，以持續驗證恢復計畫而無需人工介入，有助於維持對恢復目標的信心並符合法規遵循要求。

如需詳細資訊、請參閱 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/manage.html#manage-replication-plans["管理複寫計畫"^]。



===== Kubernetes 工作負載的排程探索

透過排程探索，您可以控制 Disaster Recovery 掃描新 Kubernetes 工作負載的頻率。您可以設定探索時間間隔，以符合您的作業節奏，減少不必要的系統負荷，同時確保及時偵測並保護新的工作負載。

如需詳細資訊、請參閱 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/sites-add.html#edit-the-site-and-customize-the-discovery-schedule["自訂探索排程"^]。



==== VMware 更新



===== 保護空資料存放區

災難復原功能現在允許為不包含任何虛擬機器（VM）的資料儲存配置保護。之後新增到資料儲存中的任何 VM 都會被 Disaster Recovery 自動偵測，此時您需要提供link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/sites-add.html#add-subnet-mapping-for-a-vcenter-site["對應資訊"^]。

將空資料存放區新增至保護計畫時，您必須先從保護計畫中移除該資料存放區，然後再執行容錯移轉，或執行強制容錯移轉。如需詳細資訊、請參閱 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/failover.html["容錯移轉至遠端站台"^]。



===== 將先前排除的資料存放區還原至保護計畫

先前被排除在保護範圍之外的資料存放區現在可以重新納入保護範圍。這項增強功能在長期管理和更新災難恢復組態時提供了更大的靈活性。

如需詳細資訊、請參閱 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/manage.html["管理站台、資源群組和複寫計畫"^]。



===== 加快基於 VMFS 的資料存放區容錯移轉速度

災難復原功能提升了基於 VMFS 的資料存放區的容錯移轉效能。主機匯流排介面卡 (HBA) 重新掃描和資料存放區掛載作業現在可以並行執行，從而縮短整體容錯移轉時間，並最大限度地減少還原事件期間的停機時間。

如需容錯移轉的詳細資訊，請參閱 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/use/failover.html["透過災難復原進行容錯移轉"^]。



=== 勒索軟體抵禦能力



==== 增強保護策略的靈活性

Ransomware Resilience 現在提供了更靈活的勒索軟體偵測策略選擇。以前，所有勒索軟體防護策略都必須包含加密偵測。現在，您可以啟用僅包含使用者行為偵測的勒索軟體防護策略。

保護策略仍然可以同時包含加密偵測和可疑使用者行為偵測。任何保護策略還可以新增封鎖檔案副檔名的功能。

如果停用現有原則的部分保護策略，則在先前原則版本下建立的任何現有警示只能標記為「已解決」，而不能標記為「已忽略」。例如，如果您的保護策略包含加密偵測和可疑使用者行為偵測，但您停用了加密偵測，則只能解決該工作負載的現有加密警示。

如需詳細資訊，請參閱link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/rp-use-protect.html["使用 NetApp Ransomware Resilience 保護策略來保護工作負載"^]。



==== SOAR 劇本增強功能

勒索軟體恢復能力 SOAR 劇本現在提供了在阻止使用者或將磁碟區離線後的補救步驟。如果該操作被判定為並非勒索軟體威脅，您現在可以將磁碟區重新上線並解除對使用者的封鎖。

link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/reference-soar.html["整合 SOAR 劇本"^]。



=== Cloud Volumes ONTAP



==== AWS 中 Cloud Volumes ONTAP 的新 EC2 執行個體支援

Cloud Volumes ONTAP 9.16.1 及更高版本支援 M6id 和 M6idn EC2 執行個體系列，適用於 AWS 中新建和現有的單節點及高可用性（HA）配對部署。與第五代通用系統相比，這些第六代執行個體提供更快的處理能力、更強大的網路功能和更高的效能。

* *M6id 實例*：兼具運算和記憶體容量，並配備高速、低延遲的本地 NVMe 儲存。所有 M6id 執行個體大小均支援 Flash Cache。這些實例非常適合資料記錄和媒體處理等工作負載。
* *M6idn 執行個體*：提供與 M6id 執行個體相同的本機 NVMe 儲存功能。所有 M6idn 執行個體大小均支援 Flash Cache。


我們不建議將較小尺寸的 m6id.xlarge、m6id.2xlarge、m6idn.xlarge 和 m6idn.2xlarge 執行個體類型用於高效能或記憶體密集型工作負載。對於這些工作負載，請使用具有更高核心數和更大記憶體的執行個體類型。

如需詳細資訊，請參閱 https://docs.netapp.com/us-en/cloud-volumes-ontap-relnotes/reference-configs-aws.html["AWS 中 Cloud Volumes ONTAP 支援的組態"^]。

=== Cloud Volumes ONTAP



==== AWS 中 Cloud Volumes ONTAP 和資料服務的彈性計費

NetApp 在 AWS 中為您的 Cloud Volumes ONTAP 訂閱提供了更進階的計費選項，讓您能更靈活地控制 Cloud Volumes ONTAP 容量與 NetApp 資料服務（例如 NetApp Backup and Recovery）的付款方式。過去，所有 Cloud Volumes ONTAP 及資料服務的使用情形，都是透過 AWS marketplace 中的 NetApp Intelligent Services（NIS）條目一起計費。

由於 AWS 企業折扣方案（EDP）的變更，NetApp 在 AWS Marketplace 中推出了獨立的 Cloud Volumes ONTAP 訂閱，讓您可以將 Cloud Volumes ONTAP 容量的計費與資料服務的計費分開。這種分離方式讓您更靈活地管理 EDP 抵扣額度和成本。NetApp 建議您使用獨立訂閱來更好地管理 Cloud Volumes ONTAP 的使用成本，同時在 2026 年 7 月 31 日 EDP 要求生效後，繼續使用 NIS 訂閱來支付資料服務的計費。

AWS 中的獨立 Cloud Volumes ONTAP 訂閱僅支援有效的基於容量的 Essentials 或 Freemium 許可套餐。Professional 套餐不支援。您可以選擇最適合您需求的計費方式：

* *獨立 Cloud Volumes ONTAP 訂閱（建議）*：您可以透過 AWS Marketplace 中的獨立訂閱支付 Cloud Volumes ONTAP 的使用費用，同時繼續透過 NIS 訂閱支付資料服務費用。對於大多數在 NetApp Console 中同時擁有獨立訂閱和 NIS 訂閱並與其 AWS 帳戶相關聯的客戶，這是建議的選擇。
* *合併 NIS 訂閱*：透過現有的 NIS 清單同時支付 Cloud Volumes ONTAP 和資料服務的費用。此合併方案適用於在 NetApp Console 中僅關聯了 NIS 訂閱的現有客戶。但是，NetApp 建議您設定 Cloud Volumes ONTAP 使用量的計費偏好設定，並切換到獨立訂閱，以便更好地管理成本。


根據您的訂閱類型，您將遇到以下情況：

.僅限獨立訂閱（新客戶）
如果您在 NetApp Console 中僅擁有與您的 AWS 帳戶關聯的獨立 AWS 訂閱，而沒有 NIS 訂閱，則在部署期間只能選擇 Essentials 授權包。資料服務的 *Backup and Recovery* 選項對您不可用。

.獨立訂閱和 NIS 訂閱（現有客戶）
如果您的 AWS 帳戶同時關聯了 NIS 和獨立訂閱，且您尚未透過 NetApp Console 中的 *Billing preferences* 標籤設定帳單首選項，則 Cloud Volumes ONTAP 的使用量預設仍透過 NIS 計費。對於執行 Professional 套餐的任何 Cloud Volumes ONTAP 執行個體，這也是預設設置，因為獨立清單不支援 Professional。您可以設定帳單首選項並選擇每項服務的計費方式：

* 將獨立訂閱設定為 Cloud Volumes ONTAP（包含 Essentials 套餐）的首選計費帳戶。如果您只有 Professional 套餐，可以將其轉換為 Essentials 套餐，並透過設定計費首選項來收取 Cloud Volumes ONTAP 的使用費用。我們推薦這種方法，因為它有助於您更有效地管理 Cloud Volumes ONTAP EDP 積分。
* 使用 NIS 訂閱進行資料服務計費，或透過 Professional 套餐計費 Cloud Volumes ONTAP 使用量。在部署期間，啟用 *Backup and Recovery* 資料服務，並選擇 Professional 授權套餐，以透過 NIS 訂閱計費資料服務。


.僅限 NIS 訂閱用戶（現有客戶）
如果您是已擁有與 AWS 帳戶關聯的 NIS 訂閱的現有客戶，則此說明適用。您可以繼續像現在一樣同時對 Cloud Volumes ONTAP 和資料服務進行計費。但是，NetApp 建議您切換到獨立訂閱模式，以便更好地管理 Cloud Volumes ONTAP 的使用成本：

* 在 AWS 入口網站上訂閱新的獨立清單。
* 將獨立訂閱連結至您的 AWS 帳戶。
* 將您的計費偏好設定為使用 Essentials 套餐的 Cloud Volumes ONTAP 獨立訂閱。
* （如適用）將您的 Professional 套件轉換為 Essentials 套件，並設定計費偏好設定。
* 繼續透過您的 NIS 訂閱進行資料服務計費。



NOTE: 使用多種訂閱類型或在清單之間遷移時，不會發生重複計量。

=== Cloud Volumes ONTAP



==== 改進了 Cloud Volumes ONTAP 對 Google Cloud C3 VMs 的支援

Cloud Volumes ONTAP 現在增強了對 Google Cloud C3 系列 VM 的支援，從而實現了以下附加功能：

* 在 Cloud Volumes ONTAP 部署中，您可以在 C3 系列的不同變體之間切換執行個體類型。
* 變更執行個體類型時，如果您選擇 c3-standard-44 或 c3-standard-88 及其 LSSD 變體作為目標 VM，則會自動套用 Google 網路服務出口網路層級 Tier-1。如果您切換到任何其他 C3 變體，則出口網路層級將設定為 Standard。
+

NOTE: 將出口網路層級變更為 Tier-1 會產生額外成本。請據此規劃部署和升級。

* 如果您選擇 C3 的本機 SSD (LSSD) 版本，則預設會啟用 Flash Cache。即使目標版本支援 Flash Cache，您也無法在 Cloud Volumes ONTAP 部署中切換 C3 LSSD 版本。
* 現在，您可以在 C3 VM 上為新的和現有的 Aggregate 新增 Volume。
* 您可以透過 NetApp Console 或使用 `PUT /gcp/vsa/aggregates/{workingEnvironmentId}/{aggregateName}` API 在新增磁碟區時變更 IOPS 和吞吐量設定。
